## 背景
- 現在のローカル開発手順は Docker 開発用イメージに依存しており、セットアップに時間がかかるほか、Docker Desktop が必須となるため軽作業や短時間の検証に不向きという声が上がっている。
- uv へ移行済みの依存関係管理を活かし、Python 仮想環境のみで開発できるようにすれば、オンボーディングの待ち時間を短縮しつつ CI やリリース向けコンテナとの互換性も維持できる。
- Docker 固有のボリュームマウントや権限周りのトラブルが報告されており、uv ベースのローカル開発環境を公式化することでドキュメントの簡素化とサポート負荷の低減が期待できる。

## 変更内容
- ローカル開発環境の推奨手順を Docker コンテナから uv が管理する Python 仮想環境へ切り替える。
- `openspec/project.md` とエージェント向けガイド（`openspec/AGENTS.md`／`openspec/AGENTS.ja.md`）を更新し、開発環境の標準が uv 仮想環境であることを明記する。
- 新しい「developer-environment」スペックを追加し、uv 仮想環境のセットアップ手順と検証コマンドを要件化する。
- 既存ドキュメント（README など）とツールチェーン手順を見直し、Docker を前提としない流れへ調整する。

## 影響
- 開発者は Docker を使用せずに uv 単体で環境を構築できるようになるが、Docker ベースのワークフローが必要な場合はオプションとして残す方針を検討する必要がある。
- CI やリリース配送は引き続き Docker コンテナで行うため、ローカル限定の変更であってもビルドに影響が出ないかの検証が必要。
- ドキュメントや自動化スクリプトから Docker 専用コマンドを削除する場合、既存ユーザーへの移行ガイドを提供しなければ混乱を招く可能性がある。
- uv 仮想環境でのコマンド実行手順を統一することで、ラッパースクリプトやチートシートの更新が求められる。
